# Vir Space Metrics Collection Runbook

This guide explains exactly how to run experiments and collect metrics from the instrumented app.

## How to read this runbook

- **Manual** = you or your teammates must do this action.
- **Automatic** = app/analyzer handles this after you trigger the workflow.

## What this runbook covers

- Workspace management metrics (create/join/auth latency and success)
- Workspace synchronization metrics (whiteboard/editor propagation)
- File transfer metrics (latency, throughput, integrity outcomes)
- Resilience metrics (resync after join/leave/reconnect disruptions)

## Log and output locations

- **Automatic:** Raw per-peer logs are written to:
  - `.metrics/<run_id>/peer_<peer_file_id>.ndjson`
- **Automatic:** Analysis outputs are written to:
  - `.metrics/<run_id>/analysis/`

> `run_id` is derived from room ID as `room-<roomId>`.

## 1. One-time setup

1. **Manual:** From repository root, install dependencies:

```bash
npm run install:all
```

2. **Manual:** Start the client on each laptop:

```bash
npm run dev --prefix client
```

3. **Manual (optional):** Start standalone signaling server:

```bash
npm run dev --prefix server
```

## 2. Start a test run

1. **Manual:** Pick a **unique room ID** for this run (example: `exp01_baseline_2026_04_04`).
2. **Manual:** Host creates room with that room ID.
3. **Manual:** Guests join the same room.
4. **Automatic:** Metrics logging initializes when room flow starts.
5. **Manual:** Locate the run folder at `.metrics/room-<roomId>/` from repository root.

## 3. Perform actions (no manual stopwatch required)

### A) Workspace management latency + success

**Manual:** Run each case 5-10 times (fresh room ID per trial):

1. Create room (host)
2. Join room with correct password (guest)
3. Join with incorrect password (negative auth test)
4. Join non-existent room ID (not-found case)

### B) Workspace synchronization latency

**Manual:**

1. Whiteboard: draw 10 short strokes, ~1 second apart
2. Editor: type one sentence, then pause >= 3 seconds (editor has batching)
3. Repeat from different peers

### C) File transfer latency + throughput + integrity

**Manual:** For each file size (1 MiB, 10 MiB, 100 MiB recommended):

1. Share file
2. Accept/download on receiver
3. Repeat each size 5 times
4. Include both text and binary files

### D) Resilience and resync

**Manual:**

1. Late join after edits
2. Unexpected disconnect during active session
3. Reconnect and rejoin
4. Mid-transfer disconnect, then recover

## 4. Gather logs after each run

1. **Manual:** On each laptop, open the run folder from filesystem at `.metrics/room-<roomId>/`.
2. **Manual:** Collect all `peer_*.ndjson` files for that run.
3. **Manual:** Place them in the same run directory on one machine for analysis.

## 5. Generate metrics automatically

1. **Manual:** From repository root, run:

```bash
npm run analyze:metrics -- room-<your-room-id>
```

2. **Manual (alternative):** You may also pass a direct path:

```bash
npm run analyze:metrics -- .metrics/room-<your-room-id>
```

## 6. Files generated by analyzer

**Automatic:** Under `.metrics/<run_id>/analysis/` after running the analyzer:

- `room_latency.csv`
- `transfer_metrics.csv`
- `workspace_rtt.csv`
- `resync_metrics.csv`
- `metrics_summary.json`

## 7. Recommended experiment hygiene

1. **Manual:** Use a new room ID for each trial batch.
2. **Manual:** Keep baseline runs on stable LAN/Wi-Fi.
3. **Manual:** Run degraded-network scenarios as separate run IDs.
4. **Manual:** Do at least 5-10 repetitions per scenario.
5. **Manual:** Report median, p95, min/max, and success percentages.
